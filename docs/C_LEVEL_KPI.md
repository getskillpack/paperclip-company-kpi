# KPI для C-level: что может задавать CEO и как это ложится на плагин

Документ для [XDE-106](/XDE/issues/XDE-106). Цель — **не заменить** performance review, а дать board **один экран**: деньги компании + **явные цели по ролям**, которые CEO выставляет и при необходимости обновляет.

## 1. Типовой набор метрик (по ролям)

| Роль | Примеры KPI | Источник факта в Paperclip (сейчас / позже) |
|------|-------------|---------------------------------------------|
| **CTO / инж. лид** | Лимит месячного burn по агентам; SLA на разблокировку `blocked`; throughput (закрытые тикеты / неделя) | Burn: события cost + бюджеты агентов (фаза C плана). Throughput: позже — запросы к issues API или ручной actual |
| **CMO / маркетинг** | Публикаций / неделя; охват (ручной); конверсия лендинга (ручной / внешняя аналитика) | MVP: **ручной actual** в цели |
| **CFO / финансы** (если выделены) | План расходов по статьям; отклонение от плана | Ручной журнал плагина + цели |
| **COO / доставка** | WIP по критическому пути; доля тикетов без stale-check > N дней | Позже — агрегаты по issues; MVP — ручной actual |
| **CEO (самоконтроль)** | Runway proxy, # эскалаций на board, фокус (активные эпики) | Частично ручной |

## 2. Принцип в продукте

1. **Цель задаётся человеком** (CEO/board): метка роли, название KPI, числовой target, единица, период (строка `YYYY-MM` или произвольная метка квартала).
2. **Факт в MVP** — в основном **ручной** (`actualValue`), чтобы не блокировать релиз на новых API host.
3. **Следующие итерации:** подтягивать actual из уже существующих данных плагина (rollup cost по агенту/роли) или из расширений host (`agents.read`, issues), см. `IMPLEMENTATION_PLAN.md` фаза C/E.

## 3. Хранение и API плагина

- State key: `executive_kpi_targets_v1` (scope company), массив записей `ExecutiveKpiTargetV1`.
- Data: поле `executiveTargets` в ответе `companyKpiDashboard`.
- Actions: `upsertExecutiveKpiTarget`, `deleteExecutiveKpiTarget` (по `id`).

Подробности схемы — в типах `src/kpi-types.ts`.
